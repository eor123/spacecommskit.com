#System Overview

The SCK-915 Explorer Kit is a three-layer system. Understanding how the layers interact is essential before modifying any component.

┌─────────────────────────────────────────────────────────┐ │ Windows PC SCK Ground Station (C# .NET 8 WinForms) │ └──────────────────────────┬──────────────────────────────┘ │ COM5 · 115200 baud · 3.3V TTL · ESP framing ┌──────────────────────────▼──────────────────────────────┐ │ Board 0001 CC1110 · Original OpenLST firmware (relay) │ └──────────────────────────┬──────────────────────────────┘ │ RF 437 MHz · 7.4 kbaud · 2-FSK + FEC ┌──────────────────────────▼──────────────────────────────┐ │ Board 0004 CC1110 · Custom firmware · PICO_MSG handler │ └──────────────────────────┬──────────────────────────────┘ │ UART0 · P1_5/P1_4 · 115200 baud · ESP framing ┌──────────────────────────▼──────────────────────────────┐ │ Raspberry Pi Pico MicroPython · main.py · camera + SD │ └─────────────────────────────────────────────────────────┘

All communication uses the OpenLST ESP frame format. Board 0001 relays packets transparently — it does not need custom firmware. Only board 0004 requires the custom PICO_MSG handler.

Never modify files inside open-lst/. All customization goes in radios/openlst_437/. This keeps your changes clean and makes upstream updates safe to apply.

#ESP Protocol

All communication between every component uses the same ESP (Embedded Serial Protocol) frame format defined in open-lst/common/uart.h.

// ESP frame on the wire
0x22  0x69  <length>  <payload bytes...>

// ESP_START_BYTE_0 = 0x22
// ESP_START_BYTE_1 = 0x69
// length = number of payload bytes (1–251)
// ESP_MAX_PAYLOAD = 251 bytes

// IMPORTANT: length must be ≥ 1. Zero-length payloads are
// silently rejected by the UART RX ISR (c < 1 check).

#Packet Structure

The ESP payload carries an OpenLST command packet. The first 6 bytes are always the command header.

// Payload layout (from commands.h)
typedef struct command_header {
    hwid_t           hwid;     // 2 bytes LE — destination hardware ID
    uint16_t         seqnum;  // 2 bytes LE — sequence number
    uint8_t          system;  // 1 byte — MSG_TYPE_RADIO_IN (0x01) for commands
    radio_msg_no_t   command; // 1 byte — opcode
} command_header_t;

// Full command (header + up to 245 bytes of data)
typedef struct command {
    command_header_t header;
    uint8_t  data[251 - sizeof(command_header_t)];
} command_t;
ByteFieldValue
0–1HWIDLittle-endian destination hardware ID
2–3SequenceLittle-endian sequence number (16–64000)
4System0x01 = MSG_TYPE_RADIO_IN (required for processing)
5OpcodeCommand opcode byte
6+DataCommand-specific payload (optional)
System byte
The system byte at offset 4 must be 0x01 (MSG_TYPE_RADIO_IN). Packets with any other value are forwarded rather than processed by the command handler. This is how relay works — board 0001 forwards packets addressed to other HWIDs over RF.

#CC1110 Firmware — Introduction

The CC1110 firmware is written in C and compiled with SDCC (Small Device C Compiler) for the 8051 architecture. The CC1110 has very limited resources:

ResourceTotalNotes
Flash32 KBBootloader uses ~7 KB; radio app ~6 KB; rest available
XDATA RAM3.3 KBAlready heavily used by UART buffers and command buffers
IRAM256 bytesStack and fast variables only
Clock26 MHz8051 core — much slower than modern MCUs

Memory constraints are the most common source of issues when extending the firmware. The XDATA allocations in the existing code leave very little margin.

#Build System

The build uses GNU Make with mingw32-make on Windows. The entry points are:

# Top-level Makefile
include radios/openlst_437/Board.mk   # your config (processed first)
include open-lst/Build.mk             # core build rules

# Build targets
mingw32-make openlst_437_radio        # build radio app hex
mingw32-make openlst_437_bootloader   # build bootloader hex (rarely needed)

Board.mk — Your Build Config

# radios/openlst_437/Board.mk
RADIOS += openlst_437
BOOTLOADERS += openlst_437
openlst_437_DIR := $(dir $(lastword $(MAKEFILE_LIST)))

# Source files compiled for radio app
openlst_437_SRCS := \
    $(openlst_437_DIR)/board.c

# CFLAGS — -DCUSTOM_COMMANDS enables the hook in commands.c
openlst_437_CFLAGS := \
    -DCUSTOM_BOARD_INIT \
    -DCUSTOM_COMMANDS \
    -DUART0_ENABLED=1 \
    -I$(openlst_437_DIR)

# Bootloader disables UART0 to save space
openlst_437_BOOTLOADER_CFLAGS := -DUART0_ENABLED=0
Always Clean + Build
SDCC does not reliably detect all dependency changes. Always use Clean + Build (or delete the specific .rel file) after any source change. Stale object files are the most common cause of mysterious behavior.

Cleaning Safely on Windows

mingw32-make clean fails on Windows because it uses Linux rm. The ground station app has a Clean button that safely deletes only radio build artifacts — files matching *.openlst_437.rel without .bl. in the name. Bootloader files (*.openlst_437.bl.rel) are preserved.

#Adding Custom Commands (CC1110)

The OpenLST firmware has a built-in extension point for custom commands. When -DCUSTOM_COMMANDS is in your CFLAGS, any opcode not handled by the standard switch statement falls through to your custom_commands() function.

Step 1 — Declare in board.h

// radios/openlst_437/board.h
#define CUSTOM_COMMANDS 1
#include "commands.h"

// Forward declaration — implementation in board.c
uint8_t custom_commands(const __xdata command_t *cmd,
                        uint8_t len,
                        __xdata command_t *reply);

Step 2 — Implement in board.c

// radios/openlst_437/board.c
#include "commands.h"
#include "uart0.h"     // UART0 = Pico interface (P1_4/P1_5)

#define PICO_MSG         0x20
#define PICO_TIMEOUT     20000   // inner loop count
#define PICO_TIMEOUT_OUT 10      // outer loop count (~2s total)

uint8_t custom_commands(const __xdata command_t *cmd,
                        uint8_t len,
                        __xdata command_t *reply) {
    uint8_t  payload_len;
    uint8_t  rx_len;
    uint16_t timeout;
    uint8_t  outer;

    switch (cmd->header.command) {

        case PICO_MSG:
            payload_len = len - sizeof(cmd->header);
            uart0_send_message((__xdata uint8_t *) cmd->data, payload_len);

            rx_len = 0;
            for (outer = 0; outer < PICO_TIMEOUT_OUT; outer++) {
                timeout = PICO_TIMEOUT;
                while (timeout > 0) {
                    rx_len = uart0_get_message((__xdata uint8_t *) reply->data);
                    if (rx_len > 0) goto pico_done;
                    timeout--;
                }
            }
            pico_done:
            if (rx_len > 0) {
                reply->header.command = common_msg_ack;
                return sizeof(reply->header) + rx_len;
            }
            reply->header.command = common_msg_nack;
            break;

        default:
            reply->header.command = common_msg_nack;
            break;
    }
    return sizeof(reply->header);
}

#Critical Firmware Rules

These rules were learned through extensive debugging. Violating any of them produces subtle, hard-to-diagnose failures.

RuleReason
Never initialize reply->header.hwid, seqnum, or systemcommands.c already sets these before calling you. Doing it again corrupts memory.
Only set reply->header.commandThis is the only field custom_commands owns.
Never use WATCHDOG_CLEAR macroThe inline assembly conflicts with SDCC register allocation in this call context and corrupts RF receive state.
Never declare large static __xdata buffersXDATA is nearly full. Write directly into reply->data which is already allocated.
Use uint8_t/uint16_t only for timeout countersuint32_t generates much larger 8051 code and causes RAM overflow crashes.
Use nested loops for large timeoutsuint16_t silently wraps at 65535. outer(10) × inner(20000) = ~2 seconds safely.
Always send ≥1 byte payloadThe UART RX ISR rejects ESP frames with length 0 (c < 1 check).
Custom commands belong in board.cSeparate .c files cause RAM overflow and linker errors due to different memory context.

// 0       reserved           1 byte

// 9–12    uart1_rx_count     uint32 LE  ← UART1 packets received
// 13      rx_mode            uint8
// 14      tx_mode            uint8

To verify UART0 TX is actually firing, use the loopback technique: connect a jumper wire between P1_4 and P1_5 on board 0004, send a PICO_MSG command, then check if uart0_rx_count incremented in the next telem response.

#ESP Framing on the Pico

All communication between the Pico and CC1110 uses ESP framing. The CC1110's uart0_send_message() automatically wraps outgoing data. The Pico must parse incoming frames and wrap its responses.

# ESP constants — must match open-lst/common/uart.h
ESP_START_0 = 0x22
ESP_START_1 = 0x69
MAX_PAYLOAD = 251

def send_esp(payload_bytes):
    """Wrap payload in ESP framing and send over UART0."""
    frame = bytes([ESP_START_0, ESP_START_1, len(payload_bytes)]) + payload_bytes
    uart.write(frame)

def recv_esp(timeout_ms=5000):
    """Parse one complete ESP-framed packet from UART0."""
    STATE_START0, STATE_START1, STATE_LENGTH, STATE_PAYLOAD = 0, 1, 2, 3
    state, payload, remaining = STATE_START0, bytearray(), 0
    deadline = time.ticks_add(time.ticks_ms(), timeout_ms)
    while time.ticks_diff(deadline, time.ticks_ms()) > 0:
        if uart.any():
            b = uart.read(1)[0]
            if state == STATE_START0:
                if b == ESP_START_0: state = STATE_START1
            elif state == STATE_START1:
                if b == ESP_START_1:   state = STATE_LENGTH
                elif b == ESP_START_0: state = STATE_START1
                else:                  state = STATE_START0
            elif state == STATE_LENGTH:
                if 1 <= b <= MAX_PAYLOAD:
                    remaining, payload, state = b, bytearray(), STATE_PAYLOAD
                else: state = STATE_START0
            elif state == STATE_PAYLOAD:
                payload.append(b)
                remaining -= 1
                if remaining == 0: return bytes(payload)
        else: time.sleep_ms(1)
    return None

#Sub-Opcode System

CC1110 opcode 0x20 (PICO_MSG) carries a one-byte sub-opcode as the first payload byte. The CC1110 strips the OpenLST header and forwards the raw payload — so payload byte 0 is the sub-opcode the Pico sees.

Sub-OpcodeNameRequest PayloadResponse
0x00PING(none)PICO:ACK
0x01READ_TEMP(none)TEMP:13.47C
0x02SNAP(none)SNAP:OK:snap_001.jpg:4617 or SNAP:ERR:reason
0x03LIST(none)LIST:file1.jpg,file2.jpg or LIST:EMPTY
0x04GET_INFOfilename bytesINFO:filename:bytes:chunks
0x05GET_CHUNKchunk_idx(2 bytes LE) + filenameCHUNK:N: + binary data

#Adding a New Pico Command

To add a new command (e.g. sub-opcode 0x07 to read a GPIO pin):

1. Add the constant

CMD_READ_GPIO = 0x07

2. Add the handler in the main loop

elif sub_opcode == CMD_READ_GPIO:
    pin_num = payload[1] if len(payload) > 1 else 0
    try:
        p = Pin(pin_num, Pin.IN)
        val = p.value()
        send_esp(f"GPIO:{pin_num}:{val}".encode())
    except Exception as e:
        send_esp(f"GPIO:ERR:{e}".encode())

3. Add a custom command in the ground station

In the Custom Commands tab, add a new command with opcode 0x20 and payload 07 <pin_number>. No code changes required — the custom commands system handles arbitrary payloads.

Response length limit
All Pico responses must fit within MAX_PAYLOAD - 1 = 250 bytes after ESP framing. For long responses (like LIST with many files), truncate gracefully.

#Protocol Layer (OpenLstProtocol.cs)

Building Packets

// Build an ESP-framed command packet
public static byte[] BuildPacket(ushort hwid, ushort seq,
                                  byte opcode, byte[]? args = null)
{
    var payload = new List<byte>
    {
        (byte)(hwid & 0xFF), (byte)((hwid >> 8) & 0xFF),
        (byte)(seq  & 0xFF), (byte)((seq  >> 8) & 0xFF),
        0x01,   // MSG_TYPE_RADIO_IN — required for processing
        opcode
    };
    if (args != null) payload.AddRange(args);
    var frame = new List<byte> { 0x22, 0x69, (byte)payload.Count };
    frame.AddRange(payload);
    return frame.ToArray();
}

Parsing Responses (RxPacket)

// RxPacket is a C# record — immutable, value-based equality
public record RxPacket(ushort Hwid, ushort SeqNum,
                        string OpName, int AckValue, byte[] RawPayload)
{
    // Auto-decodes ASCII Pico responses from ack payloads
    // RawPayload layout: hwid(2) + seq(2) + flag(1) + opcode(1) + data
    // Data starts at index 6
    public string? PicoPayload
    {
        get {
            if (RawPayload == null || RawPayload.Length < 7) return null;
            if (OpName != "ack") return null;
            byte[] data = RawPayload.Skip(6).ToArray();
            if (!data.All(b => b >= 0x20 && b <= 0x7E)) return null;
            return Encoding.ASCII.GetString(data);
        }
    }
}

All tabs follow the same pattern. Here is the minimum required to add a new tab:

1. Declare the tab page and controls

// In field declarations section of MainForm.cs
private TabPage tpMyTab = null!;

2. Create the tab page

// In the tab creation block
tpMyTab = MkTab("  My Tab  ");
tabMain.TabPages.Add(tpMyTab);

3. Build the tab contents

private void BuildMyTab()
{
    var p = tpMyTab;
    int lx = 10, row = 12;

    p.Controls.Add(MkSectionLabel("── My Section", lx, row)); row += 28;

    btnMyAction = MkButton("▶ Do Something", lx, row, 140,
        Theme.Green, Theme.PanelBack);
    btnMyAction.Click += async (s, e) => await MyActionAsync();
    p.Controls.Add(btnMyAction);
    row += 42;

    lblMyStatus = MkLabel("Ready", lx, row, 500, Theme.Gray, Theme.FontSmall);
    p.Controls.Add(lblMyStatus);
}

private async Task MyActionAsync()
{
    if (!CheckConnected()) return;
    ushort hwid = ActiveHwid;
    ushort seq  = IncSeqNum();

    FlushRxQueue();
    WritePacket(OpenLstProtocol.BuildPacket(hwid, seq, 0x20,
        new byte[] { 0x00 }));   // PING sub-opcode
    var pkt = await WaitForReply(hwid, seq, 10000);

    if (pkt?.PicoPayload != null)
        Log($"Result: {pkt.PicoPayload}", Theme.Green);
    else
        Log("No response", Theme.Red);
}

#Parsing Binary Pico Responses

The GET_CHUNK command returns binary data prefixed with a text header. The ground station parses this by finding the second colon:

// Response format: "CHUNK:N:" + binary JPEG data
// RawPayload = hwid(2) + seq(2) + flag(1) + opcode(1) + response bytes
byte[] raw = chunkPkt.RawPayload.Skip(6).ToArray();

// Find second colon to split "CHUNK:N:" from data
int firstColon  = Array.IndexOf(raw, (byte)':');
int secondColon = firstColon >= 0
    ? Array.IndexOf(raw, (byte)':', firstColon + 1) : -1;

if (secondColon >= 0)
{
    byte[] chunkData = raw.Skip(secondColon + 1).ToArray();
    imageStream.Write(chunkData, 0, chunkData.Length);
}

The AES signature is computed as CBC-MAC over the application bytes from APP_START to SIG_START, using an all-zero IV. All Explorer Kit boards ship with all-0xFF signing keys.
